# Balance — Functionality Audit & Fix Plan (July 2026)

*Produced 2026-07-11 from a five-track deep audit of the codebase (tasks, productions,
secondary pages, data layer/schema, intake/notifications/collab) plus a live click-through.
Companion to `docs/ROADMAP.md` — this doc is the "what's actually broken or missing"
deep-dive; ROADMAP stays the running phase tracker.*

---

## Part 1 — BROKEN: bugs found (things that look like they work but don't)

These are ranked by user-facing damage. Each is independently fixable.

### P0 — Data silently lost

1. **Task fields that never persist.** `expectationsNote`, `completionNote`, and
   `assignedBy` are set by the UI and displayed prominently, but have **no DB column**
   and are dropped by `taskToRow` (`src/lib/data/tasks.ts:103-118`). They survive only
   until refresh. Users are typing notes into a void.
   *Fix:* add three columns + map them. ~1 hour + migration.

2. **Comment photos are broken end-to-end.** TaskCard uploads the photo to storage
   (`TaskCard.jsx:97-107`) and passes `photoStoragePath`/`photoName`/`authorName` —
   `AppContext.addComment` (`AppContext.jsx:454-481`) discards all three, `createComment`
   inserts body-only, and the renderer reads `c.photoUrl` **which is never set anywhere**
   (`TaskCard.jsx:452-462`). Photos upload, then vanish. Dead upload path + orphaned files.
   *Fix:* `photo_storage_path`/`photo_name` columns on `task_comments`, stop discarding,
   render from `storage_path` via signed URL. ~half day.

3. **Task status history is write-only.** The `task_status_history` table is fully built
   (trigger, RLS, realtime) but the client **never reads it** — `rowToTask` hardcodes
   `statusHistory: []` (`tasks.ts:81`). Consequence: "Verified by X" attribution
   (`TaskCard.jsx:312-322`) works only within one session; TeamPage turnaround/on-time
   metrics are computed from `updatedAt` and acknowledged in-code as unreliable
   (`TeamPage.jsx:342-344`). The audit trail exists — surface it.
   *Fix:* `listTaskStatusHistory(taskId)` + render. Small, high value.

### P0 — Core workflow unreachable

4. **Half the task workflow can't be used.** Statuses In Progress / Needs Review /
   Blocked exist in the DB CHECK, the visual config, and a role-aware transition state
   machine (`taskStatusConfig.js:72-95`) — but **no UI ever calls it**. The only wired
   transitions are Not Started → Complete → Verified. `blockedReason` is display-only.
   The "Awaiting Verification" filter's NEEDS_REVIEW branch, TeamPage's blocked/in-progress
   buckets, and Dashboard's blocked coloring can only fire on demo data.
   *Fix:* status picker on TaskCard driven by `getValidTransitions` (already written!),
   with a blocked-reason prompt when moving to Blocked. ~half day, unlocks the whole model.

5. **No way to create a task outside a production.** TasksPage has zero creation entry
   point — confirmed live. Quick capture is the #1 daily-use PM action.
   *Fix:* "New Task" button on TasksPage with a production picker. Small.

### P1 — Wrong numbers / dead displays

6. **Damage count on production cards is always 0.** Card filter reads `a.damageFlag`
   (`ProductionsPage.jsx:401`) but addons set `damaged` (`models.js:307`). Dead badge.
7. **"Complete" means different things on different pages.** Dashboard progress bars and
   Analytics count VERIFIED-only (`DashboardPage.jsx:220`, `AnalyticsPage.jsx:49`);
   TeamPage counts COMPLETE||VERIFIED (`TeamPage.jsx:332`). Same task, different truth.
   Pick one definition (suggest: COMPLETE||VERIFIED = "done", VERIFIED = "signed off")
   and centralize in `taskStatusConfig.js` (helpers `isTaskDone` already exist, unused).
8. **Dashboard "View all" under My Tasks navigates to `/productions`** instead of
   `/tasks` (`DashboardPage.jsx:157`).
9. **Feedback `resolutionNote` is rendered but never editable** — admin controls offer
   only status + delete (`FeedbackPage.jsx:245-279`). Dead display field.
10. **AnalyticsPage violates Rules of Hooks** — admin `<Navigate>` early-return sits
    above six `useMemo`s (`AnalyticsPage.jsx:26`). Role change mid-session can crash.
11. **`created_by` never populated on tasks**; `is_demo` unmapped for tasks; contractor
    model fields (`skills`, `dayRate`, `weeklyRate`, `emergencyContact`, …) have **no DB
    columns** and are silently dropped (`contractors.ts:70`); stale TS types
    (`ProductionType` union contradicts free-form reality, `productions.ts:10-11`).
12. **Breadcrumb labels out of sync** with sidebar (`/coming-soon` → "Coming Soon" vs
    "Roadmap"; `/todos`, `/feedback` unmapped) (`Breadcrumbs.jsx:17-29`).

### P1 — Schema integrity

13. **`is_admin()` is referenced by 5 DELETE policies across migrations but defined in
    none of them.** Either it was created by hand in the dashboard (drift — out of
    version control) or a clean `db reset` fails. Verify against live DB; add a
    migration defining it either way.
14. **Optimistic updates never roll back.** Every `updateX` merges locally, then on
    server error only `console.error`s — local state diverges until reload
    (`AppContext.jsx:397,434,520`). Compounds the no-toast problem.
15. **Storage read RLS is bucket-wide** — any authenticated user can read any path in
    any bucket (`phase3:39`, acknowledged in-code). Acceptable short-term (paths are
    UUIDs), fix post-identity-unification.

### P2 — Cosmetic/minor

- `PROD_TYPE_TINT` keyed to legacy type strings; nearly all cards fall to fallback tint.
- Gantt day-width computed against hardcoded 800px (`SchedulePage.jsx:295`).
- Stage grouping matches exact string `'In-House (Orbital Studios)'` (`SchedulePage.jsx:169`).
- Dead code: legacy date regexes + `CONCERN_KEYWORDS` in `intakeUtils.js`; the obsolete
  duplicate voice-memo recorder in `InstructionPackage.jsx:85-97` (ephemeral blob URLs,
  superseded by `VoiceRecorder.tsx` + transcribe function); `updateTaskInstructionPackage`
  exported but never called (`AppContext.jsx:670-681`).
- ComingSoon delete confirm references nonexistent `item.title` (harmless fallback).

## Part 2 — FOUNDATIONS: known structural debt (confirmed, quantified)

Unchanged from ROADMAP but the audit sharpened the blast radius:

1. **Split identity model** (string ids vs auth UUIDs) is the root blocker under:
   notifications RLS (`using(true)` — anyone can read anyone's), per-assignment
   production visibility (abandoned → all-published-visible-to-all-crew), comment author
   attribution, `assignee_id`-scoped RLS, and the notification trigger's self-notify
   check misfiring. **Do this first; five other fixes are gated on it.**
2. **localStorage triple** (Gear/To-Dos/Feedback) — all three present as multi-user
   shared tools and are single-browser. Feedback is the sharpest trap: crew submit bugs
   admins will literally never see. To-Dos' SHARED/DIRECT visibility model is cosmetic.
3. **Silent write failures** — no toast system; combined with #14 above (no rollback),
   a failed save both hides the error AND keeps showing the phantom edit.
4. **Zero automated tests** — `intakeUtils`, `roadmapUtils`, and the mappers are pure
   and trivially unit-testable; the auth flow deserves one Playwright smoke.
5. **Notifications = exactly one event** (task assignment, via DB trigger). Nothing on
   comments, status changes, mentions, due dates, publishes.
6. **Unbounded queries** — every `list*` is `select('*')` with no limit except
   notifications. Fine at studio scale; time-bomb as data accumulates.
7. **Pending:** `phase6h` migration not yet run against live Supabase.

## Part 3 — Monday.com gap analysis

What Monday has that Balance doesn't, filtered for what actually matters to an
8-person VP studio, with the implementation path given Balance's architecture.

### Tier 1 — core PM primitives Balance is missing (build these)

| Monday feature | Balance today | Path to build |
|---|---|---|
| **Updates & @mentions** (conversation on every item) | Comments on tasks only; photos broken; no mentions | Fix comment persistence (P0 #2), add `production_comments` table (same pattern), @mention autocomplete from USERS/profiles → row in `notifications`. The notifications table + realtime bell already exist. |
| **Activity log per item/board** | `task_status_history` written, never read; nothing else audited | Read the existing table (P0 #3); add a generic `activity_events` table (entity_type, entity_id, actor, verb, meta jsonb) written from AppContext mutation hub (`mutateProduction` is a single choke point — one place to instrument). Render per-production feed tab. |
| **Kanban view** | Flat card list only; status machine already written but unused | Board columns = 6 existing statuses, drag between columns calls `getValidTransitions`-validated update. All visual config exists in `taskStatusConfig.js`. This is the cheapest "wow" in the codebase. |
| **My Work** (one inbox across boards) | "Mine" filter on TasksPage; stats scattered on TeamPage | Dedicated `/my-work`: assigned tasks + owned milestones + due to-dos + mentions, grouped by date bucket (To-Dos page already has the bucketing logic to reuse). |
| **Multiple views: Calendar / Timeline** | Schedule page is read-only Gantt; no task calendar | Month calendar with tasks+milestones by due date; make Schedule bars draggable (drag = update start/end). The Gantt data adapter (`dataSource.js`) already derives everything. |
| **Automations** ("when X then Y") | Five hardcoded DB triggers | Don't build a rules UI yet. Ship the 4 automations that matter as code: due-soon reminder (scheduled edge function → notifications), overdue escalation, status-change notify (watcher = assignee + creator), mention notify. A `pg_cron` or Vercel cron hitting one edge function covers scheduled ones. |
| **Notifications that reach you** (email/push, not just bell) | Bell only, one event type | Expand event coverage first (cheap, DB triggers), then daily email digest via edge function + Resend (free tier covers 8 people trivially). |
| **Item templates / duplicate** | None; every production starts blank | "Duplicate production" (copy structure not status) + save-as-template of milestone/task sets. The intake wizard already builds type-specific starter tasks (`intakeUtils.js:532-556`) — generalize that table into user-editable templates. |
| **Subitems (subtasks)** | Flat | `parent_task_id` self-FK + indent in card/board views. Moderate. |
| **Global search / Ctrl+K** | None ("8 nav items deep, no way to jump") | Command palette (cmdk pattern): fuzzy over productions/tasks/contractors/pages in client state — all data is already loaded into AppContext, so v1 needs no backend at all. Cheap. |

### Tier 2 — valuable, build after Tier 1

| Monday feature | Notes for Balance |
|---|---|
| **Workload/capacity view** | The math already exists in the prototype (`dataSource.js:265-280` computes overcommitted per person/day). Promote it out of the 3D prototype into a boring, useful per-person utilization strip on Team/Schedule. |
| **Dependencies** | `dependencies[]` on milestones is modeled and dead. Wire milestone dependencies first (visual chain in Timeline), task dependencies later. |
| **Forms** (public intake) | Balance's intake wizard *is* a form — a shareable client-facing "brief us" link that feeds the same parser is a natural extension (needs guest-safe RLS or an edge function inbox). |
| **Dashboards (configurable)** | Add date-range + export to Analytics before making widgets configurable. Include drill-through (chart → filtered task list). |
| **Guest/client access** | Share-token read-only production view (the Handoff Summary is the seed of this — it already compiles the client-facing packet). Bigger lift: new role + RLS. |
| **Time tracking** | Skip classic timers. A "days on production" derived view from assignments probably serves a VP studio better. Revisit on demand. |
| **Custom fields** | Skip. Fixed schema is a feature at this team size; jsonb `custom_fields` if a real need appears. |
| **Docs (workdocs)** | Skip. Production Bible already covers the structured-notes need better than a generic doc editor would. |
| **Recurring tasks** | To-Dos are the natural home (daily ops); add `recurrence` on todos when they move to Supabase. |
| **AI features** | Balance already has a better hook than Monday's generic AI: Tier 2 Claude intake (vision on screenshots → structured production draft). Deferred by choice; the stubs and fallback architecture are ready. |

## Part 4 — THE PLAN

Sequenced so every phase ships user-visible value while unblocking the next.
Estimates assume focused Claude-assisted sessions.

### Phase 0 — Stop the bleeding (2–3 days) ✅ ship immediately
Bug-fix blitz, no architecture changes:
1. Persist the lost task fields (expectations/completion/assignedBy) — migration + mappers.
2. Fix comment photos + author names end-to-end.
3. Wire the full status workflow UI (picker driven by existing `getValidTransitions`,
   blocked-reason prompt). Read `task_status_history` for attribution + TeamPage metrics.
4. "New Task" from TasksPage. Fix Dashboard "View all" link.
5. One "done" definition, centralized; fix damage-count field; Analytics hooks fix;
   editable resolution note on Feedback; breadcrumb sync; define `is_admin()` in a
   migration (after verifying live DB).
6. **Toast system** (from ROADMAP issue #3) + rollback-on-error for updates — do it in
   this phase because every later phase writes more.

### Phase 1 — Foundations (≈2 weeks)
1. **Identity unification** (the 2–3 day migration; five features are gated on it).
2. **Port Gear / To-Dos / Feedback to Supabase** (1 day each; patterns exist).
   To-Dos' visibility model becomes real RLS.
3. Run `phase6h`; then tighten notifications SELECT + per-assignment production
   visibility (both unblocked by #1).
4. Test baseline: Vitest on intakeUtils/roadmapUtils/mappers + one Playwright smoke.
5. Paid Supabase project + custom domain (kills the auto-pause trap).

### Phase 2 — Monday-parity core (≈4 weeks, each item independently shippable)
Ordered by value-per-effort:
1. **Kanban board view** on TasksPage (status machine already exists).
2. **Global search / command palette** (client-state fuzzy search; cheap).
3. **My Work page** (assigned tasks + owned milestones + to-dos + mentions).
4. **Comments everywhere + @mentions + notification expansion** (production comments,
   mention→notification, comment/status-change triggers).
5. **Activity feed** (`activity_events` table + per-production tab; instrument
   `mutateProduction` chokepoint).
6. **Interactive Schedule** (drag-to-reschedule bars, drag milestone diamonds; container-
   width fix comes free).
7. **Production templates + duplicate** (generalize intake starter-task tables).
8. **Automations v1 as code**: due-soon reminders, overdue escalation, daily email
   digest (edge function + cron + Resend).
9. **Subtasks** (`parent_task_id`).

### Phase 3 — Differentiators (ongoing, pick by demand)
- Workload/capacity strip (promote prototype math into Team/Schedule).
- Milestone dependencies UI (field already modeled).
- Client share-links (read-only production view grown from Handoff Summary).
- Tier 2 Claude intake (vision on screenshots; heuristics stay as fallback).
- Calendar sync (Google), budget fields + estimate-vs-actual, Gear DB v2,
  bundle diet, Sentry, configurable Analytics + export.

### Explicitly deprioritized (decided, not forgotten)
Custom fields, workdocs, classic time tracking, recurring production tasks, a no-code
automation builder — wrong size for an 8-person studio; revisit only on real demand.
